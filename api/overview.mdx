Grim exposes a public Java API that allows other plugins and extensions to integrate deeply with the anticheat. You can listen for flag events, inspect per-player data, provide a custom config backend, manage alert state, and register custom message placeholders — all without modifying Grim's source.

The API is split across two artifacts:

- **GrimAPI** — the public interface library, published separately at [github.com/GrimAnticheat/GrimAPI](https://github.com/GrimAnticheat/GrimAPI). Depend on this at compile time.
- **Grim** (the plugin itself) — provides the runtime implementation. Declare it as a `depend` or `softdepend` in your `plugin.yml`.

## The singleton entry point

All API access starts from the `GrimAPI` singleton:

```java
GrimAPI.INSTANCE
```

`GrimAPI` holds every internal manager. For external plugin use, Grim exposes a curated surface through `GrimExternalAPI`, which implements the `GrimAbstractAPI` interface:

```java
GrimAbstractAPI api = GrimAPI.INSTANCE.getExternalAPI();
```

You can also retrieve the same object from Bukkit's `ServicesManager`:

```java
GrimAbstractAPI api = Bukkit.getServicesManager()
    .load(GrimAbstractAPI.class);
```

<Note>
`GrimAPI.INSTANCE.getExternalAPI()` and the `ServicesManager` entry both return the same `GrimExternalAPI` instance. Prefer the `ServicesManager` approach when writing soft-depend code so your plugin can still load even if Grim is absent.
</Note>

## What you can do

| Feature | How |
|---|---|
| Listen for flag, command, and reload events | `api.getEventBus()` |
| Inspect per-player data | `api.getGrimUser(player)` |
| Manage alert / verbose state | `api.getAlertManager()` |
| Provide a custom config backend | `api.reload(ConfigManager)` |
| Register custom message placeholders | `api.registerVariable(...)` |
| Resolve a plugin to a `GrimPlugin` | `api.getGrimPlugin(this)` |
| Read the current Grim version | `api.getGrimVersion()` |
| Read the current server tick | `api.getCurrentTick()` |
