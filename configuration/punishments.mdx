title: "Punishment configuration"
description: "Configure punishment groups, commands, thresholds, and violation decay in punishments/en.yml."
---

Grim's punishment system is defined entirely in `plugins/GrimAC/punishments/en.yml`. Rather than assigning punishments per-check, Grim uses **punishment groups** — named buckets that aggregate violations from multiple related checks and trigger commands when cumulative violations cross defined thresholds.

## File structure

```yaml punishments/en.yml
Punishments:
  GroupName:
    remove-violations-after: 300   # seconds
    checks:
      - "CheckName"
      - "!ExcludedCheck"
    commands:
      - "threshold:interval command"
```

Every key under `Punishments:` is a named group. You can add, remove, or rename groups freely.

---

## Checks list

The `checks` list controls which Grim checks contribute violations to this group. Matching is case-insensitive and uses substring matching against both the check's primary name and any alternative names.

```yaml
checks:
  - "Simulation"    # matches all checks whose name contains "Simulation"
  - "GroundSpoof"
  - "Timer"
  - "!BadPacketsN"  # exclude BadPacketsN even if it would otherwise match
```

<Info>
  Prefix a check name with `!` to **exclude** it from a group that would otherwise match it. This is useful when a broad term like `"BadPackets"` would pull in a specific check you want to handle differently.
</Info>

---

## Commands format

Each entry in the `commands` list uses the format:

```
threshold:interval command
```

| Part | Description |
|------|-------------|
| `threshold` | Minimum violation count before this command becomes eligible to run. |
| `interval` | How often (in violations) the command repeats after the threshold. Use `0` to execute exactly once. |
| `command` | The command or special keyword to run. |

**Examples:**

```yaml
commands:
  # Alert staff when violations reach 100, then again every 40 violations after that.
  - "100:40 [alert]"

  # Log to database at the same cadence.
  - "100:40 [log]"

  # Send a Discord webhook alert at 100 violations, then every 100 after.
  - "100:100 [webhook]"

  # Kick the player exactly once when violations hit 200.
  - "200:0 kick %player% Movement violation limit reached."
```

---

## Special commands

Instead of a server command, you can use one of Grim's built-in action keywords:

| Keyword | Action |
|---------|--------|
| `[alert]` | Broadcast an in-game alert to all players with the `grim.alerts` permission. |
| `[log]` | Write the violation to the configured database (see `history` in `config/en.yml`). |
| `[webhook]` | Send a Discord embed to the configured webhook URL. |
| `[proxy]` | Forward the alert to all other servers on your BungeeCord/Velocity network. |

<Note>
  `[proxy]` requires `alerts.proxy.send: true` in `config/en.yml` and the BungeeCord plugin messaging channel enabled on your proxy.
</Note>

## Alert format placeholders

The `[alert]` message is controlled by `alerts-format` and `alerts-format-proxy` in `punishments/en.yml` (or you can set it globally). The following placeholders are supported:

| Placeholder | Description |
|-------------|-------------|
| `%player%` | The flagged player's name. |
| `%check_name%` | Display name of the check that flagged. |
| `%vl%` | Current violation count for this check within the group. |
| `%verbose%` | Extra debug information from the check (offsets, states, etc.). |
| `%description%` | The check's description string. |
| `%experimental%` | Replaced with the experimental symbol (default `*`) if the check is experimental, otherwise empty. |
| `%prefix%` | Grim's configured message prefix. |

Default alert format (from `PunishmentManager.java`):

```
%prefix% &f%player% &bfailed &f%check_name% &f(x&c%vl%&f) &7%verbose%
```

## remove-violations-after

```yaml
remove-violations-after: 300
```

Violations older than this many **seconds** are automatically removed from the group's violation map. This prevents old flags from indefinitely inflating a player's violation count after they stop cheating.

<Tip>
  Set `remove-violations-after` lower (e.g. `60`) for checks that produce many short-burst violations to avoid legitimate players accumulating punishments from intermittent false positives.
</Tip>

---

## test-mode

```yaml
test-mode: false
```

When `test-mode: true`, `[alert]` messages are sent **only to the flagged player** instead of broadcasting to all staff. This lets you test punishment thresholds in production without spamming your staff channel.

    ```yaml

      remove-violations-after: 300
      checks:
        - "Simulation"
        - "GroundSpoof"
        - "Timer"
        - "TimerLimit"
        - "NoFall"

        - "100:40 [alert]"
        - "100:40 [log]"

        - "100:100 [proxy]"
    ```

## Adding a kick punishment

To kick a player after a sustained period of violations, add a console command entry to any group:

```yaml
Simulation:
  commands:
    - "100:40 [alert]"
    - "100:40 [log]"
    - "200:0 kick %player% Kicked for illegal movement."
```

The `200:0` format means: execute **once** when the violation count reaches 200. Replace `kick` with any server command available on your platform.

<Warning>
  Avoid setting kick thresholds too low on checks with high natural variance (e.g. Simulation, BadPackets). Start with alerts, monitor verbose output, and raise or lower thresholds based on observed false-positive rates before enabling kicks.
</Warning>
